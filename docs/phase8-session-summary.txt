PHASE 8 SESSION SUMMARY
MileagePlus Deal Finder — Order-Level MILP Basket Optimizer with Two-Phase Async Response
Date: 2026-03-25
==================================================

OBJECTIVE
---------
Implement the order-level MILP basket optimizer in the Julia engine with a
two-phase async response (greedy immediate + MILP polled), wire /basket and
/basket/status/:job_id endpoints in both Julia and the Node bridge, and add
a Basket tab to the React frontend where the user enters a shopping list and
sees first a greedy solution and then the MILP-optimized assignment when it
completes. Per-order bonuses (flat_tiered, per_order_flat) are correctly
modeled as order-level decision variables, enabling the MILP to find bonus
consolidation opportunities that the per-item greedy solver misses.

IMPORTANT NOTES FOR FUTURE SESSIONS
--------------------------------------
1. Julia binary: /usr/local/bin/julia (Julia 1.12.5)
   Always run with: JULIA_PKG_SERVER=""

2. JuMP v1.30.0 and HiGHS v1.22.2 resolved without version conflicts.
   Both packages were already in Project.toml from a prior session.

3. HiGHS WAS available in the test environment. All MILP tests ran
   successfully without skipping. The try/catch guard in test_phase8.jl
   exists but was not triggered.

4. Test 3 (bonus threshold consolidation) CONFIRMED the MILP improves
   on greedy: 740 miles (MILP) vs 450 miles (greedy), a 64.4% improvement.
   The MILP correctly consolidates two $60 items at BestBuy to trigger the
   $100-threshold flat_tiered bonus of 500 miles.

5. App.jsx final line count: 1100. BasketTab was split into a separate
   component file (frontend/src/BasketTab.jsx, ~250 lines) to keep
   App.jsx under control. Only 3 targeted edits were made to App.jsx:
   import, tab button, and rendering.

6. BASKET_JOBS is an in-memory Dict in the Julia server process. Jobs
   are lost on server restart. This is acceptable for a personal tool.

7. The MILP uses order-level decision variables: x[i,o] (item to order),
   y[o] (order active), z[t] (bonus tier active), s[o] (order subtotal).
   Each order has a fixed (retailer, path) pair. This is the correct
   formulation per v3-spec section 5.1.

8. Cross-retailer stacked paths (buy GC at retailer A, shop at retailer B)
   are NOT modeled in the basket MILP. Only same-retailer stacked paths
   are included. This keeps the candidate order count manageable. Extending
   this would require O(N^2) candidate orders for stacked paths.

9. The greedy solver sorts items by miles-per-dollar descending before
   assigning, so when budget is tight, the highest-value items are
   prioritized.

10. tax_rate in the basket endpoint is passed directly to SpendVector.
    The bridge converts percentage to decimal (same as /api/score and
    /api/rank). When calling Julia directly, pass decimal (e.g., 0.08).

11. The MILP time_limit defaults to 28 seconds, leaving 2 seconds of
    buffer within the 30-second bridge timeout. The solver attribute
    mip_rel_gap is set to 0.01 (1% optimality gap tolerance).
