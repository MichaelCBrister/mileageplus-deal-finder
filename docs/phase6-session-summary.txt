PHASE 6 SESSION SUMMARY
MileagePlus Deal Finder — Playwright Scraper with Snapshot Grouping and Portal-to-DB Pipeline
Date: 2026-03-25
==================================================

OBJECTIVE
---------
Build a Playwright-based scraper that authenticates with the MileagePlus
Shopping portal, scrapes retailer base rates, bonus offers, and T&C text,
writes everything to the database under a single scrape snapshot, and chains
with the Phase 5 parser to populate tc_rules and bonus_offers with structured
data. A single command produces a complete snapshot that the Julia scoring
engine can read.

IMPORTANT NOTES FOR FUTURE SESSIONS
--------------------------------------
1. Julia binary: /usr/local/bin/julia (Julia 1.12.5)
   Always run with: JULIA_PKG_SERVER=""

2. The scraper ran against MOCK DATA, not the live portal. Playwright Chromium
   could not be downloaded in the cloud environment. The mock (portal-mock.js)
   returns 3 retailers matching seed data. When running on desktop with
   Cowork, set MILEAGEPLUS_USERNAME and MILEAGEPLUS_PASSWORD and Playwright
   will attempt live portal scraping.

3. Live portal DOM selectors are BEST-EFFORT and untested. The scraper.js
   has selectors for common Valuedynamx/Collinson portal structures:
     - Retailer list: a[href*="/b__"], .store-card a, .merchant-link
     - Rate: .earn-rate, .miles-rate, .rate-value, .store-rate
     - T&C: .terms-conditions, .tc-text, .store-terms
     - Bonus: .bonus-offer, .promo-text, .special-offer
   These MUST be verified with SCRAPER_DEBUG=1 on first live run.

4. ANTHROPIC_API_KEY was NOT available. T&C and bonus parsing was skipped.
   The scraper writes raw text to tc_rules and bonus_offers with empty
   inclusions/exclusions and confidence=0.0. The Julia engine treats
   confidence < 0.8 as risk_class=uncertain (portal miles still included
   with warning). Bonus tiers are empty so bonus_miles=0.

5. TIMESTAMP FORMAT: The snapshot.js formats timestamps without milliseconds
   or Z suffix (e.g., "2026-03-25T21:11:24" not "2026-03-25T21:11:24.549Z").
   This matches the Julia engine's dateformat"yyyy-mm-ddTHH:MM:SS" parser
   in snapshot_age_hours(). The audit log (request-log.js) still uses full
   ISO8601 with milliseconds since it's not read by Julia.

6. KNOWN ISSUE: The Julia engine's snapshot_age_hours function cannot parse
   ISO8601 timestamps with milliseconds or Z suffix. This was worked around
   in the scraper by stripping milliseconds/Z from timestamps. If a future
   phase generates timestamps in a different format, the Julia function will
   need to be updated.

7. When ANTHROPIC_API_KEY is available and parsing runs, the tc_rules and
   bonus_offers rows are updated in-place (UPDATE by retailer_id + snapshot_id).
   The inclusions field must include ALL eligible categories per Phase 5
   reconciliation rule: a category not in inclusions returns excluded.

8. The one-per-day guard checks for snapshots with status IN ('complete',
   'partial') started on the current UTC date. To re-scrape, either wait
   until the next UTC day or manually delete today's snapshot from the DB.

9. Database initialization: rm -f db/mileageplus.db && bash db/init.sh
   Must delete existing DB first because INSERT OR IGNORE won't update
   existing rows from prior seed runs.
